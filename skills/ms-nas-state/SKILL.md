# ms NAS State

## 概述

用本地 `cli-anything-ms` 获取 NAS 当前系统状态，并整理成简洁的中文结论。

## 工作流

### 1. 确认 CLI 可用

```bash
pip install cli-anything-ms
```

- 如果安装失败，直接转述安装失败原因，停止后续步骤。

### 2. 调用 NAS 信息接口

- 固定执行下面这条命令，不要改成插件接口：

```bash
cli-anything-ms --json system nas-info
```

- 不要要求用户手动拼接参数。
- 不要调用旧的 `zspace_service_assistant` 插件接口。

### 3. 理解返回结果

- 返回结果是 NAS 设备列表，通常每个元素代表一台 NAS 实例。
- 默认总结全部设备；如果用户指定设备名、实例名或型号，只回答匹配设备。
- 如果命令返回空列表，明确告诉用户当前没有拿到 NAS 状态数据。
- 如果返回结构不符合预期，明确说明“已调用成功，但 NAS 状态数据结构异常”。
- 每个设备可能包含 `raw` 字符串；可尝试按 JSON 解析，用于补充硬盘、存储空间、系统版本、UPS 配置等扩展信息。
- 如果 `raw` 解析失败，不要中断回答，只说明部分扩展硬件细节不可用。

### 4. 提取重点信息

- 优先总结这些信息：
    - 设备名称、厂商、型号、在线状态、系统版本、运行时长
    - CPU 占用、温度
    - 内存总量、已用量、使用率
    - 存储空间容量、已用空间、使用率
    - 磁盘型号、容量、温度、状态、是否待机
    - 网络 IP、上下行速度
    - 风扇转速
    - UPS 厂商、型号、连接状态、电池或供电信息

### 5. 总结并回复

- 先给一句总体状态概述，例如“当前整体运行正常”。
- 多台 NAS 时逐台简洁总结。
- 再补充关键指标摘要，尽量像正常汇报，不要写成字段清单。
- 如果用户只问某一类指标，例如 CPU、内存、磁盘温度，就优先回答该部分，同时保留一句整体判断。

## 异常提示规则

- 如果 NAS 状态不是在线，要显式提示。
- 如果 CPU、磁盘或 SSD 温度明显偏高，要显式提示关注。
- 如果内存或存储空间使用率较高，要显式提示。
- 如果某块磁盘状态不是正常状态，要点名指出。
- 如果接口调用失败，例如认证失败、请求超时或后端返回错误，直接转述失败原因，不要编造系统状态。

## 注意事项

- 不增加历史对比。
- 不做趋势分析。
- 不主动给出修复建议清单，除非用户继续追问。
- 输出以中文总结为主，不要暴露插件返回字段名或 JSON 结构。
